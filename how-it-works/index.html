The Decision Workflow Lifecycle

From business logic to auditable execution in six steps.

Architecture: Ingest → Design → Compile → Execute → Audit

Ingest

Data arrives from your systems

Databases, APIs, event streams. Redis Data Integration or lightweight adapters normalize it.

Object Library

Typed, validated inputs/outputs

No loose JSON. No string parsing. Every field validated before processing.

Flow Design

Visual editor for business logic

Compose rules, decisions, and AI checkpoints. No code required.

Compile

Deterministic bytecode

Flows become executable graphs. No runtime interpretation. Zero surprises.

Execution

Redis Streams orchestrate

State machines execute. Decisions are immutable. Audit trails automatic.

Artifacts

Results + Evidence

Decisions written to RedisJSON, downstream systems, and live dashboards.

Why this matters: Each layer is intentional. No ambiguity. No surprises at 3 AM.

From Design to Executable

Flows are not interpreted at runtime. They are compiled.

What compilation does

  • Validates all paths (no dead code, no missing branches)
  • Optimizes execution (reorder steps, cache queries)
  • Generates audit metadata
  • Creates deterministic hashes (same flow = same hash)
  • Outputs bytecode or optimized dependency graph

Why it matters

  • Zero runtime surprises
  • Impossible to accidentally change logic mid-execution
  • Compiled flows are versioned and immutable
  • Rollback is trivial (switch version flag)

Deterministic, Auditable Decision Execution

How it works

  1. Trigger: Event arrives (API, queue, scheduled). Written to Redis Stream.
  2. Consumer Group: Execution engine reads, guarantees at-least-once processing.
  3. State Machine: Compiled flow executes. Each step writes to Streams (immutable log).
  4. Queries: Flow queries RedisJSON for inputs, cached rules, AI models.
  5. Output: Decision written to Streams and stored in RedisJSON.

Why Streams instead of Kafka?

  • In-memory, sub-millisecond, still persistent
  • Latency matters more than geo-distributed failover
  • If you run Redis, no additional tool needed
  • Complete history stored in Redis; no separate audit DB

Where AI Fits (And Where It Doesn't)

AI goes here

  • Classification: "Is this fraud?"
  • Extraction: "What is customer intent?"
  • Ranking: "Rank tickets by urgency"
  • Recommendation: "What product to suggest?"
  • Content Moderation: "Policy violation?"

AI does NOT go here

  • Writing business logic: Encode rules instead
  • Making binary decisions: Use rules + AI confidence
  • Generating audit-critical text: Generate from rules

Pattern: AI as Advisor. Your business logic decides. AI informs. The combination is auditable.
