The Decision Workflow Lifecycle

From business logic to auditable execution in six steps.

Architecture: Ingest → Design → Compile → Execute → Audit

Ingest

Data arrives from your systems

Databases, APIs, event streams. Redis Data Integration or lightweight adapters normalize it.

Object Library

Typed, validated inputs/outputs

No loose JSON. No string parsing. Every field validated before processing.

Flow Design

Visual editor for business logic

Compose rules, decisions, and AI checkpoints. No code required.

Compile

Deterministic bytecode

Flows become executable graphs. No runtime interpretation. Zero surprises.

Execution

Redis Streams orchestrate

State machines execute. Decisions are immutable. Audit trails automatic.

Artifacts

Results + Evidence

Decisions written to RedisJSON, downstream systems, and live dashboards.

Why this matters: Each layer is intentional. No ambiguity. No surprises at 3 AM.

Data Integration: The RDI Question

You have 20 years of data in Oracle, Postgres, Snowflake. A flow needs applicant credit history. Does it call the database directly? No—that's blocking, slow, and stateless.

Option A: Redis Data Integration (Enterprise)

If you run Redis Enterprise, RDI provides CDC connectors to your legacy databases. Data syncs to Redis in real-time. Flows query Redis (in-memory, sub-millisecond). Old data becomes a local copy. New data updates it continuously.

Cost: Part of Redis Enterprise licensing. Setup: Configuration-driven.

Option B: Lightweight Adapters (OSS)

You don't need RDI if you're building once. We write thin integration layers: REST API translators, scheduled syncs, event stream consumers (Kafka → Redis).

Cost: Our time to build (part of engagement). Your infrastructure only.

Result: Both approaches normalize data into Redis. Flows don't know or care which you chose. The architecture is agnostic to integration.

The Visual Design Problem and How We Solve It

No enterprise has 12 months to teach engineers "decision flow language." Your business stakeholders understand decisions—they make them daily. They just can't code.

We inverted the problem. Instead of "write code that expresses decisions," we use visual flow design: nodes for rules, branches for conditions, AI checkpoints, loops, parallel execution. You design. We compile to bytecode. No runtime interpretation. No ambiguity.

Why compilation matters

  • Interpreted workflows fail at runtime (in production)
  • Compiled flows fail at design time (in testing)
  • We validate all paths, detect dead code, catch type mismatches
  • Generate deterministic hashes (same flow = same hash)
  • For compliance: "which version decided this?" → specific bytecode hash

Versioning

Each flow becomes a composable, versionable unit of business logic. You change a threshold, you get a new version (v2.1.1). Old decisions stay reproducible under v2.1.0. No "this rule used to be X."

From Design to Executable

Flows are not interpreted at runtime. They are compiled.

What compilation does

  • Validates all paths (no dead code, no missing branches)
  • Optimizes execution (reorder steps, cache queries)
  • Generates audit metadata
  • Creates deterministic hashes (same flow = same hash)
  • Outputs bytecode or optimized dependency graph

Why it matters

  • Zero runtime surprises
  • Impossible to accidentally change logic mid-execution
  • Compiled flows are versioned and immutable
  • Rollback is trivial (switch version flag)

Deterministic, Auditable Decision Execution

How it works

  1. Trigger: Event arrives (API, queue, scheduled). Written to Redis Stream.
  2. Consumer Group: Execution engine reads, guarantees at-least-once processing.
  3. State Machine: Compiled flow executes. Each step writes to Streams (immutable log).
  4. Queries: Flow queries RedisJSON for inputs, cached rules, AI models.
  5. Output: Decision written to Streams and stored in RedisJSON.

Why Streams instead of Kafka?

  • In-memory, sub-millisecond, still persistent
  • Latency matters more than geo-distributed failover
  • If you run Redis, no additional tool needed
  • Complete history stored in Redis; no separate audit DB

Compliance and Auditability

Compliance is not a feature you bolt on. It's the architecture. Regulatory queries become trivial.

"Show me all loan denials from Q4 where credit score was < 650."

SELECT decision FROM redis_decisions
WHERE flow = "underwriting:v2.1.0"
  AND outcome = "denied"
  AND credit_score < 650
  AND date BETWEEN Q4_start AND Q4_end

Returns: 347 decisions. For each, you have: applicant ID, exact inputs, exact rules that fired, exact model output, timestamp, duration. Evidence complete.

Where AI Fits (And Where It Doesn't)

AI goes here

  • Classification: "Is this fraud?"
  • Extraction: "What is customer intent?"
  • Ranking: "Rank tickets by urgency"
  • Recommendation: "What product to suggest?"
  • Content Moderation: "Policy violation?"

AI does NOT go here

  • Writing business logic: Encode rules instead
  • Making binary decisions: Use rules + AI confidence
  • Generating audit-critical text: Generate from rules

Pattern: AI as Advisor. Your business logic decides. AI informs. The combination is auditable.
